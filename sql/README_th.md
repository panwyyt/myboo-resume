### 💻 แอบดูตัวอย่างโค้ดสักหน่อย (คลิกขยายดูได้นะ)

<details>
<summary><b>1. คำสั่ง SQL สายพื้นฐาน</b> (<code>basic_command.sql</code>)</summary>

```sql
-- สร้างตารางและตั้งข้อจำกัด (Constraints) นิดหน่อย
CREATE TABLE members (
    id SERIAL PRIMARY KEY,
    fullname TEXT NOT NULL,
    department TEXT,
    salary NUMERIC(10, 2),
    join_date DATE DEFAULT CURRENT_DATE
);

-- ลองดึงข้อมูลพร้อมใส่ตัวกรอง
SELECT fullname, salary FROM members WHERE salary >= 35000;

-- ลองอัปเดตและลบข้อมูลดูบ้าง
UPDATE members SET salary = 36000.00 WHERE id = 11;
DELETE FROM members WHERE id = 1;
```

</details>

<details>
<summary><b>2. วิเคราะห์ข้อมูลสายลึก</b> (<code>advanced_commnd.sql</code>)</summary>

```sql
-- จัดกลุ่มหมวดหมู่ด้วยเงื่อนไข CASE WHEN
SELECT animalname, animalage,
       CASE 
           WHEN animalage LIKE '%year%' THEN 'Adult/Senior'
           WHEN animalage LIKE '%month%' OR animalage LIKE '%week%' THEN 'Baby/Kitten'
           ELSE 'Adult'
       END AS age_group
FROM animals LIMIT 20;

-- จัดลำดับแรงค์ Rank โดยผ่านระบบ Window Functions
SELECT speciesname, animalname, animalage,
       RANK() OVER(PARTITION BY speciesname ORDER BY animalage DESC) as age_rank
FROM animals WHERE animalname != ''
ORDER BY speciesname, age_rank LIMIT 30;
```

</details>

## 🧰 สกิลเทคนิคที่ผมอยากเอามาโชว์

ขอสรุปแบบรวบรัดเลยว่า ในโปรเจกต์นี้ผมได้งัดเอาสกิล SQL อะไรมาใช้งานบน **PostgreSQL** บ้าง:

- **SQL พื้นฐานแบบชิลๆ (CRUD):** คำสั่งทั้ง `CREATE`, `INSERT`, `UPDATE`, `DELETE`, `DROP`, `SELECT`, กรองข้อมูลด้วย `WHERE`
- **วิเคราะห์ข้อมูลแบบแอดวานซ์:**
  - Window Functions (`RANK()`, `LEAD()`, `LAG()`) สำหรับดึงและจัดการลำดับข้อมูลเนียนๆ
  - การเล่นเงื่อนไข (`CASE WHEN`) จัดกลุ่มข้อมูลตามอิงสถานการณ์ได้แบบไดนามิก
  - จับกลุ่มและคัดกรองแบบเป็นเซ็ต (`GROUP BY`, `COUNT`, `HAVING`)
  - ใช้ Common Table Expressions (CTEs - ใช้คู่กับ `WITH`) ทำให้อ่านพวกคำสั่งยาวๆ ง่ายขึ้น ดูสะอาดตาขึ้นเยอะ
- **การจัดการดาต้าเบส:** การเซ็ต `PRIMARY KEY` แล้วก็ดึงเอาข้อมูล `CSV` ก้อนโตเข้ามาผ่านคำสั่ง `COPY`
- **สถาปัตยกรรมดาต้าเบส:** ความเข้าใจตัว Entity-Relationship (ER) Diagrams, การมองความสัมพันธ์ PK/FK ให้ออก, รวมไปถึง schema normalization (แอบเล่าไว้ข้างล่างด้วยนะ)
- **จำลองเซิร์ฟเวอร์ด้วย Docker:** อันนี้เอาไว้กางฐานข้อมูลแบบส่วนตัว (เช่น **MS SQL Server**) ผ่าน Docker เพื่อสร้างสภาพแวดล้อมจำลองการทดสอบของตัวเองแบบเร็วทันใจ

## 📖 เรื่องสนุกๆ "เบื้องหลังความพยายาม" ของผม

### 1. ก้าวออกจากคอมฟอร์ทโซนฝั่ง Front-end

จำได้เลยว่าตอนเริ่มก้าวเข้าสู่วงการทำงานเทสต์ซอฟต์แวร์ใหม่ๆ สายตาผมมองไปแค่ในส่วนที่มองเห็นได้ หรือก็คือพวกหน้าบ้าน (Front-end) ตลอด ผมเคยคิดนะว่า แค่เรากดแบบฟอร์มให้กรอกข้อมูล พอกดส่งแล้วมันเด้งว่า "เรียบร้อย" ขึ้นมา แค่นี้ก็แสดงว่างานผมจบแล้ว

แต่พอเวลาเปลี่ยนไป ความสงสัยมันชักกวนใจ ผมเริ่มถามตัวเองว่า: **"ตกลงไอ้ปุ่มที่ผมกดส่งเมื่อกี๊เนี้ย ข้อมูลมันไหลไปอยู่ตรงไหนต่อล่ะ?"**

นั่นแหละฮะ จุดเริ่มต้นที่พาผมดำดิ่งลงมาสู่หลังบ้านของระบบที่เรียกว่า "ฐานข้อมูล" (Database) ทันที

### 2. เมื่อผมโดนพลังของ SQL ตกเข้าให้เต็มๆ

ผมตระหนักได้ดีเลยว่า แค่รอดูตัวอักษร Success จากฝั่งหน้าบ้านอย่างเดียว มันยังเรียกได้ไม่เต็มปากเลยว่านี่คือการตรวจสอบที่รัดกุม ผมอยากได้วิธีการที่เราสามารถเข้าไปยืนยันด้วยตาตัวเองได้เลยว่า... "ข้อมูลทุกตัวที่ลูกค้ากรอกมา มันมาถึงที่นี่ได้อย่างถูกต้อง และถูกเซฟเอาไว้อย่างดีแม่นยำนะ"

เพื่อคว้าสกิลนั้นมาครอง ผมลุยศึกษาเรื่อง **SQL** แหลกเลยฮะ มันเลยกลายมาเป็นเครื่องมือคู่กายที่ผมใช้เปิดประตูดาต้าเบสเข้าไปแอบเช็คความถูกต้องของข้อมูลแบบไม่ให้มีอะไรมาหลอกตาได้เลย

### 3. เลเวลอัป: จับเรื่อง Schema ให้แม่นและโยงความสัมพันธ์ให้ได้

ยิ่งเรียนยิ่งรู้ว่า จะเป็นนักเทสต์ดาต้าเบสที่ดี แค่เขียนสั่ง CRUD (เอาข้อมูลเข้า ลบข้อมูลออก) อย่างเดียวไม่ได้กินหรอก เราต้องมีสกิลเข้าใจพวกตัว **Schema** (พิมพ์เขียวดาต้าเบส) และมองให้ออกว่า ตารางแต่ละตารางมันค่อยๆ เชื่อมกันผ่านเบาะแสอย่าง **Primary Keys (PK)** และ **Foreign Keys (FK)** ยังไง!

ตอนลองอ่านพวกลายแทง Entity-Relationship Diagram (ER) ก็สนุกเอาเรื่องนะฮะ อย่างเวลาทำความเข้าใจตาราง ผมใช้วิธีตั้งคำถามคุยกับตัวเองขำๆ แบบนี้:
สมมุติผมกำลังจะหาความสัมพันธ์ว่าตาราง **นักศึกษา** มันจะไปพ่วงติดกับตาราง **คณะวิชา** หยั่งไงดี... ผมก็ลองถามใจดู

- *เออนี่... เด็ก 1 คนส่วนใหญ่เค้าเรียนได้กี่คณะ?* -> (ก็มักจะได้แค่ 1 คณะไง)
- *แล้วคณะ 1 คณะ มันรับเด็กเข้าได้กี่คนล่ะ?* -> (โอโห รับได้หลายคนสิ)
- **บทสรุปของผมก็เลยออกมาเป็น:** ในเมื่อ 1 คณะมันยัดนักศึกษาลงไปได้เยอะขนาดนั้น ความสัมพันธ์ลักษณะนี้มันก็เลยชัดเจนว่าเป็นรูปแบบ **1:N (One-to-Many)** ครับ ง่ายๆ แค่นี้เลย!

### 4. ช็อตเบิกเนตร "อะฮ้า!" ตอนเดินห้าง

เรื่องสถาปัตยกรรมของดาต้าเบสที่เคยยากๆ มันก็สว่างวาบมาเข้าใจเอาตอนที่ผมลองเอามันไปผูกไว้กับระบบตะกร้าช้อปปิ้งออนไลน์นั่นแหละครับ

แรกเริ่มเดิมทีผมมโนเอาเองเลยนะว่า ถ้าระบบช้อปปิ้งออนไลน์แบบนี้มีฐานข้อมูล อย่างมากก็คงตั้งตารางมาสัก 3 อันก็พอ: เค้าก็คงมี **ตารางลูกค้า**, **ตารางสินค้า**, แล้วก็น่าจะจบที่ **ตารางคำสั่งซื้อ** (Order) แค่นี้แหละ

แต่คุณพระ! พอได้เจาะลึกลงไปดูระบบสเกลใหญ่ มันมีจุดที่ผมแบบ อะฮ้า! ร้องอ๋อขึ้นมา คือผมถึงบางอ้อว่าระบบการช้อปปิ้งของจริงมันจะง่ายๆ แค่ 3 ขาไม่ได้หรอก อย่างน้อยระบบต้องงอกตารางที่ 4 มาช่วยแน่ๆ นั่นก็คือตารางหัวบิลที่ชื่อ **Order Header** ครับ ตารางเจ้านี่แหละที่มีจุดประสงค์สำคัญในการคอยกวาดเอาไอเทมต่างๆ ที่ลูกค้าอุตส่าห์กวาดลงตะกร้าพร้อมๆ กันในรอบบิลนั้น รวบยอดให้อยู่ในใบเสร็จใบเดียวไปเลย จะได้ไม่สับสน

พอผมเก็ตเรื่องจุดเชื่อมเหล่านี้ปุ๊บ สไตล์การเทสต์ก็อัปเกรดไปเลยฮะ การที่เราดูโครงสร้างตารางขาดแต่แรกว่ามันควรเชื่อมโยงยังไง มันช่วยเซ้นส์ในการทดสอบขั้นสุด ให้เราจินตนาการได้ว่าระบบจะรับมือกับข้อมูลผู้ใช้งานยังไง ช่วยสกัดดาวรุ่งดักทางพวกรอยรั่วหรือสถาปัตยกรรมที่พังก่อนที่มันจะกลายร่างเป็นบั๊กโผล่มาทักทายฮะ!
