# 🚀 พอร์ตโฟลิโอการทดสอบ API ด้วย Postman

สวัสดีครับ! ยินดีต้อนรับเข้าสู่อีกมุมหนึ่งของพอร์ตโฟลิโอเน้นหนักด้าน API Testing ของผมเองครับ!

ในฐานะที่ผมรับบทเป็นสายคุมเส้นแบ่งตายของ Quality Assurance (QA) เป้าหมายสูงสุดของผมเลยก็คือการชำแหละซอฟต์แวร์ให้มั่นใจว่ามันต้องใช้งานได้จริงแบบเป๊ะเว่อร์ตั้งแต่ข้างในยันข้างนอกครับ จริงอยู่ที่การออกไปเทสต์พวกลายเส้นการคุยกับผู้ใช้งานบนหน้าปุ่ม User Interface (UI) ก็มีสัดส่วนที่สำคัญเหลือเกิน แต่ผมก็กล้าพูดได้เต็มปากเลยว่า... การลงลึกงัดกระดองเทสต์ "เลเยอร์ข้อมูลหลังบ้าน" หรือพวกท่อเชื่อมข้อมูลอย่างพวก APIs มันสำคัญยิ่งยวดไม่แพ้กันเลยครับ

เพราะลองคิดดูสิฮะ ถ้าท่อน้ำหล่อเลี้ยงข้อมูล (หรือตัว API นั่นแหละ) มันแข็งแรง โครงสร้างถูกปรับแต่งมาอย่างชาญฉลาดและปลอดภัย... มันก็จะช่วยปัดเป่าพวกความผิดพลาดมหาศาล หรือจุดบั๊กโง่ๆ ไม่ให้ลามทะลุโผล่ไปโชว์หราต่อหน้าต่อตาผู้ใช้งานหน้า UI ได้อีกเป็นกองเลยล่ะครับ!

ในพอร์ตโฟลิโอชิ้นนี้ ผมเลยขอเชิญเครื่องมือสามัญประจำบ้านอย่าง **Postman** ออกมาวาดลวดลาย เพื่อพิสูจน์ให้เห็นว่าผมสามารถเดินดุ่มเข้าไปพูดคุยส่งสัญญาณกับตัว APIs โดยตรง แกะลายแทงความแม่นยำของผลลัพธ์ และงัดสกิลจัดการกระบวนท่าเทสแบบอัตโนมัติ ให้ออกมาโชว์หน้าจอสำเร็จรูป—ตั้งแต่ก่อนกระบวนการตอกไข่สร้างหน้าบ้าน UI จะถูกงัดซักฟีเจอร์ขึ้นมาเขียนด้วยซ้ำ!

## 🙏 ขอขอบคุณแหล่งขุมทรัพย์ข้อมูลจำลองตัวทดสอบ

ผมคงต้องกราบขอบพระคุณพื้นที่ทรายสภาพแวดล้อมดีๆ ที่ผมดึงเอามายำและฝึกทดสอบจัดเต็มกันในพอร์ตผลงานชุดนี้ครับ ซึ่งเขาใจดีให้เราแวะเข้าไปโชว์ของโชว์สกิลได้เต็มที่เลย:

- **[JSONPlaceholder](https://jsonplaceholder.typicode.com/)**: เว็บจำลองเซิร์ฟเวอร์ยอดยุค ที่ผมท้าประลองไว้ใช้โชว์เทคนิคเบสิกพื้นฐานสร้างงาน อ่าน แก้ไข อย่างเรียบง่าย
- **[Restful-Booker](https://restful-booker.herokuapp.com/)**: เว็บจำลองฟังก์ชันงานอันละเอียดอ่อน ที่มีคอนเซปจองโรงแรมแอบยากนิดๆ เพื่อให้ผมต้องประลองตั้งด่านป้อมรับมือพวก กุญแจรหัสเข้าสู่ระบบ โทเคน และมิติเรื่องความปลอดภัยการอนุญาต

## 🎯 ผมมาทดสอบเพื่อหาอะไร?

ที่ตั้งใจลุกมาร่างโปรเจกต์ชุดนี้ ก็เพราะอยากจะลงพื้นโชว์แนววิธีชีวิตกระบวนท่าที่ผมทำประจำตอนตรวจเทสต์ในฝั่ง APIs ครับ ซึ่งนี่คือผลสรุปบางส่วนว่า... ในแต่ละวันผมตลุยอะไรแบบนี้ยังไงบ้าง:

### 1. ทดสอบการโอนข้อมูลแบบเบสิก (ตระกูลท่าพื้นฐาน CRUD)

แน่นอนว่าก่อนจะวิ่งก็ต้องหัดเดินก่อน! ผมไล่เปิดเช็คจังหวะการรับส่งพื้นฐานตั้งแต่ การสูบข้อมูล (GET), เสกสร้างข้อมูลกระรอกกระแต (POST), จับเปลี่ยนแปลงข้อมูลเดิม (PUT/PATCH), ยันไล่สาดทิ้งซบลบมันซะ (DELETE) เรียลๆ ครับ ผมลงมือกรอกหลักเกณฑ์ยิงสคริปต์สกัดชี้เป้า ว่าตัวเซิร์ฟเวอร์จะต้องยอมเด้งคืนพร้อมค่าสเตตัส Status Codes ที่เพอร์เฟค และพ่วงแนบข้อมูลโครงร่างกลับมาให้ตรงสเปคสวรรค์ทุกกระเบียดนิ้วเท่านั้น

![Basic API Test Result](boo-basic-api/runResult1.png)
![Basic API Details](boo-basic-api/runResult2.png)
*(นี่คือผลพลอยได้ที่ฟาดรายงานมาเขียวขจี ครอบคลุมท่าทดสอบจัดการข้อมูลกระบวนท่าทั้งหมดแบบสบายๆ)*

### 2. โมคอัพปั้นข้อมูลจำลองและวางกลยุทธ์สคริปต์รันอัตโนมัติ (Pre-request & Test Scripts)

เพราะเรื่องอะไรจะยอมนั่งทำเรื่องเบื่อๆ วนซ้ำให้เมื่อยนิ้วล่ะครับ! ผมก็อัดสกิลเพิ่มระบบ JavaScript พ่วงฝังในร่าง Postman ซะเลย ให้มันปั่นไหลทำงานเป็นกระบวนการของมันสิฮะ:

- **การเปิดก่อนให้สคริปต์ทำงานล่วงหน้า (Pre-request Scripts)**: ผมพิมพ์สคริปต์กระแทกลงไปให้ร่างชื่อคนบ้าๆ บอๆ พ่วงเรทราคาของมั่วๆ (หรือก็คือ Mock-up data) คลอดขึ้นมาไดนามิกก่อนจะกดลั่นไกระสุนจริง ส่งคำขอ रिक्वेस्ट! การดักรอแบบนี้จะทำให้ผมมีข้อมูลดิบทดสอบที่สดใหม่และไม่จำเจตลอดเวลาแหละครับ
- **การป้อนเทสสคริปต์ (Test Scripts)**: แถมพกผมยังสอยเขียนสูตรหลักประเมินผลคำถาม เพื่อสั่งให้ตัว Postman กดเทียบสเปคตรวจสอบค่าตอบรับจากเซร์ฟเวอร์เข้ากับเงื่อนไขของผมให้ทันควัน! ไม่ต้องมาเหนื่อยนั่งถ่างตาเล็งจ้องบรรทัดเช็คเลข Data payload ให้เวียนหัวกันแบบทีละมิลลิเมตรอีกต่อไปครับ!

![Automated Tests with Scripts](boo-advance-api/boo-advanced-Restful-Booker-runResult.png)
*(รอบสเต็ปนี้ ผมสั่งเจาะทลายป้อมปราการ ดึงกุญแจ Access Token ขึ้นมาสำเร็จ แล้วเสกโยนต่อใส่สายโซ่ไปปลดล็อกรันคำสั่งตัวต่อไปได้อย่างราบลื่นสมบูรณ์แบบ)*

### 3. ลงสนามซัดการทดสอบประสิทธิภาพเซิร์ฟเวอร์ (System Performance Testing)

บางทีแอปพลิเคชันตัวดีที่ทำงานได้ดั่งใจหวังสำหรับผู้ใช้คนเดียว... พอมันเจอโหลดผู้เล่นแห่พุ่งเข้าชนพร้อมกันเข้าหน่อยก็พับเต้นท์หนีตายเอาก็ดื้อๆ ซะงั้น! ควบคู่ไปกับการทำสอบฟังก์ชัน ผมยังกระหน่ำยิงหมวดระบบวัด Performance เพิ่ม เพื่อชำแหละนั่งดูตัว API รับกรรมตอนจำลองยูสเซอร์แห่แหนกันเหยียบส่งข้อมูลไหลแย่งเข้ามาในเวลาเดียวกัน งานนี้แหละครับ เพื่อรับประกันว่าสเตตัสกระแสการตอบรับจะไม่หน่วงอืดจนโลกแตกซะก่อน

## 💡 โบนัสสกิลที่แอบซ่อนอยู่: ความคุ้นชินกับการงัดข้อกับ Swagger

ถึงแม้โปรเจกต์กางพอร์ตมาหลักๆ ผมจะทิ้งใจเทให้กับยอดตึก Postman แต่กลไกชีวิตทดสอบ APIs ที่ถูกลืมไม่ได้เลยก็คือบอทสายโหดแห่งหน้าเอกสารคู่มืออย่าง **Swagger (OpenAPI)** นี่แหละครับ ผ่านช่วงหลักสูตรการติวเข้ม ผมได้จับถูไถบ้าระห่ำในการควานคลำ และแปลนิทานภาพวงจรโครงสร้าง APIs โดยใช้ Swagger UI เป็นเวทีกับเหล่าสรรพสัตว์ตัวป่วนสาย [Swagger Petstore](https://petstore.swagger.io/) ซะล้นหลอดเลยฮะ 

ถึงในโชว์เคสอัตโนมัติด้านบนพวกนี้อาจจะไม่ได้ตั้งใจแสดงการเต้นรอบซากลงไปเต็มๆ แต่บอกคำเดียวว่า “ผมเอาอยู่!” ถ้าต้องคว้าตัวเอกสารเจาะเนื้อหา ส่อง Endpoint เข้าไปงมตัวแปรจำเป็น หรือแกะกล่องประเมินค่าความคาดหวังผลลัพธ์ผ่านตัว Swagger แบบถึงพริกถึงขิงครับ

**แถมเป็นโปรทริคสายเทพ:** หนึ่งในกลวิธีขี้โกงยามร่ายมนต์สู้รบกับความอึดอัดของตัวคู่มือ Swagger ก็คือ... เข้าไปตีคอนฟิกข้อมูลชดเชยคำร้องมั่วๆ ใส่ UI กดยิงมันไปรอบนึงฮะ... และหลังจากนั้นก็แค่ **ปาดก๊อปไอ้โค้ดยาวเหยียดตรง cURL command ออกซะเลย**! ขอบอกว่าคุณเอามาโยนทิ้งโหลงไปในโหมด (Import > Raw text) ร่างตัว Postman ได้แบบไม่ลีลา มันจะกางบรรทัดคราฟต์หน้าตา Headers ข้อมูลเส้นทาง จุด Endpoint ดาต้าแพคเกจออกมากางแผ่ให้ครบทุกอณู... ไม่ต้องมาเสียประสาทนั่งหลังขดหลังแข็งพิมพ์ แถมพุ่งตัวไปเริ่มกระบวนการนั่งเทสต์จริงได้ไวขึ้นอีกสิบเท่า!
