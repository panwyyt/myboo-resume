# 🚀 ผลงาน JMeter Portfolio

## 🧐 Apache JMeter คืออะไร?

**Apache JMeter** คือเครื่องมือ Open Source ชั้นนำที่ออกแบบมาเพื่อการทำ **Performance Testing** และ **Load Testing** โดยเฉพาะ

**JMeter แตกต่างจาก Postman และ Playwright อย่างไร?**

- **Postman:** โดดเด่นในการทำ API Functional Testing มีอินเทอร์เฟซที่ใช้งานง่าย เหมาะสำหรับฝั่งพัฒนาและการตรวจสอบฟังก์ชันการทำงานของ API แต่ละตัว เหมาะอย่างยิ่งสำหรับการส่ง request เดี่ยวๆ หรือรัน automated flow แบบต่อเนื่อง
- **Playwright:** สร้างขึ้นสำหรับการทำ End-to-End (E2E) UI Testing โดยจำลองพฤติกรรมการใช้งานบนเว็บบราวเซอร์ของผู้ใช้จริง
- **JMeter:** คือ "ราชาแห่งการจำลองโหลด" จุดแข็งหลักคือการจำลอง Virtual Users (Threads) จำนวนมหาศาล ตั้งแต่หลักร้อยไปจนถึงหลายแสนคน เพื่อยิง request ใส่เซิร์ฟเวอร์พร้อมๆ กัน มันช่วยทดสอบว่าระบบจะล่มหรือไม่ รองรับ Throughput ได้เท่าไหร่ และมีคอขวด (Bottlenecks) ซ่อนอยู่ที่ไหน!

## 🐾 โปรเจกต์ Petstore API

โปรเจกต์นี้ใช้ **Swagger Petstore API** เพื่อจำลองสถานการณ์ Load Testing ขอขอบคุณ API สาธารณะนี้ที่เปิดให้ tester ได้ฝึกฝนที่ [https://petstore.swagger.io/#/](https://petstore.swagger.io/#/) ซึ่งเป็นพื้นที่เรียนรู้ที่ยอดเยี่ยมมาก

เรื่องราวของโปรเจกต์นี้ครอบคลุมตั้งแต่การวางโครงสร้างสคริปต์ ไปจนถึงการดีบักเชิงลึกเมื่อเจอปัญหา "Data Collision" (ข้อมูลชนกัน) ภายใต้โหลดที่หนักหน่วง!

## 🛠 เวิร์กโฟลว์และทักษะที่นำเสนอ

### 1. การสร้างสคริปต์อย่างรวดเร็วผ่าน cURL Import

แทนที่จะต้องมานั่งสร้าง HTTP Request ทีละตัว ผมใช้ฟีเจอร์ cURL Import ซึ่งช่วยลดเวลาในการสร้าง base request ลงอย่างมาก ประหยัดเวลาไปได้เยอะโดยเฉพาะกับ API ที่ซับซ้อน

![Importing cURL](petstore-api-performance/import-curl.png)

### 2. การจัดการ HTTP Header

การจัดการ header เช่น Authentication Keys, Content-Type, และ Accepts ทำผ่าน **HTTP Header Manager** เพื่อให้มั่นใจว่าทุกๆ request ที่จำลองขึ้นมีรูปแบบข้อมูลที่ถูกต้องเสมือนผู้ใช้จริงในระบบ production

![Managing Headers](petstore-api-performance/manage-header-keys.png)

### 3. การดึงค่าและส่งต่อตัวแปร (Dynamic Variable Extraction & Chaining)

ผมใช้ Post-Processors (เช่น JSON Extractor) เพื่อดึงข้อมูลสำคัญจาก response ของเซิร์ฟเวอร์และเก็บเป็นตัวแปร เพื่อส่งต่อไปยัง request ถัดไป (เช่น ดึง ID หลังจากสร้าง pet เสร็จ แล้วนำ ID นั้นไปใช้ในขั้นตอน update/delete)

![Storing Variables](petstore-api-performance/store-variables.png)

### 4. Data Parameterization & Randomization

เพื่อแก้ปัญหาข้อมูลชนกัน (Data collision) ระหว่างที่มี user เข้าพร้อมกัน ผมได้ใช้งานฟังก์ชันของ JMeter เช่น `${__Random()}` และนำไปผูกใน **User Defined Variables (UDV)** และเพื่อป้องกันการเกิด cache ผมยังได้สร้าง **"On-the-fly" Variables** (เช่น `${CURRENT_PET_ID}`) ที่จะรีเฟรช pet ID ใหม่สำหรับแต่ละ Thread ทำให้เป็นการจำลองผู้ใช้ที่แยกจากกันอย่างสมจริง

### 5. การรันสคริปต์ (GUI vs. Non-GUI)

ผมใช้โหมด GUI เป็นหลักในการตรวจสอบ สร้างสคริปต์ และดีบักเพื่อให้แน่ใจว่า flow ต่างๆ ทำงานได้ถูกต้อง

![Running Execution](petstore-api-performance/running-execution.gif)

อย่างไรก็ตาม ในช่วงที่ยิงโหลดจริง การใช้ GUI จะกินเมมโมรี่มหาศาล ผมจึงรันเทสต์ผ่านโหมด **Non-GUI (CLI)** เพื่อรักษาความแม่นยำ ซึ่งยังช่วยให้ผมดึง Interactive HTML Dashboard Reports ออกมาจาก terminal ได้โดยตรงด้วย

![Terminal Report Generation](petstore-api-performance/terminal-report-generation.png)

## 🎬 กรณีศึกษา: ภารกิจทะลวงกำแพง 404 ของโปรเจกต์ Petstore

หากมอง 3 รอบของการยิงเทสต์เป็นเรื่องราวของคนทำงานสาย QA/Tech มันจะเป็นเคสแก้ปัญหาเรื่อง performance ที่เห็นภาพชัดและสนุกมาก!

### บทที่ 1: การบุกยิงครั้งแรกและความวุ่นวาย (Run #1)

หลังจากสร้างชุดสคริปต์ CRUD (Create, Read, Update, Delete) เสร็จสมบูรณ์ เราได้ทดลองความอึดของระบบด้วยการจำลองผู้ใช้ 100 คนยิง request เข้ามาพร้อมกัน รวมทั้งสิ้น 2,500 request

**ผลที่ได้:** กราฟผลลัพธ์เป็นสีแดงเถือก! เราพบ Error Rate ที่พุ่งสูงเกือบ 20% (เฟลไป 498 ครั้ง) — ทั้งหมดติดปัญหา `404 Not Found`

**การสืบสวน:** เราเกิดปัญหาแบบคลาสสิกคือ **"Data Collision"** เนื่องจากเราเซ็ต Pet ID และ Username เป็นค่าคงที่ (Static) เมื่อผู้ใช้ 100 คนพุ่งเข้าระบบพร้อมกัน คนแรกที่เข้ามาทำสำเร็จจนไปถึงเส้น "DELETE" (ลบ) เมื่อผู้ใช้คนที่ 2-100 ตามเข้ามาเพื่ออ่านหรือลบข้อมูล pet ตัวเดียวกัน ระบบจึงตอบกลับมาตามปกติว่า: *"ฉันหามันไม่เจอ! มันถูกลบไปแล้ว!"*

**การแก้ปัญหาในรอบที่ 1:** เราพยายามแก้โดยการใช้ฟังก์ชัน Parameterization `${__Random()}` ใส่เข้าไปใน **User Defined Variables (UDV)** โดยหวังว่ามันจะสุ่ม ID ไม่ซ้ำกันให้ผู้ใช้แต่ละคน

---

### บทที่ 2: กับดัก Randomizer และปริศนาที่ยังค้างคา (Run #2)

เรายิงโหลดในรอบที่ 2 ด้วยความมั่นใจ Error Rate รวมลดลงเหลือพียง 9.68%! ชุดคำสั่ง POST, GET, และ PUT เขียวผ่าน 100%

**ผลที่ได้:** ช้าก่อน... ตอนมาถึง request `04_DELETE` ยังคงเฟลกระจายไปเกือบ 50% (เฟลไป 242 ครั้ง)

**การสืบสวน:** เราตกหลุมพรางของ JMeter! การวางเงื่อนไขฟังก์ชันสุ่มไว้ในตาราง UDV ด้านบนสุด จะสั่งให้ JMeter รันมันแค่ **"ครั้งเดียวตอนเริ่มสตาร์ทเทสต์"** มันสุ่มตัวเลขออกมา 1 ชุด (เช่น 55555) แล้วส่งแจกจ่ายตัวเลขชุดเดียวกันนั้นให้ผู้ใช้ทั้ง 100 คน! เราสร้างปัญหาเดิมซ้ำ: คนที่ไวสุดลบ 55555 ไป และที่เหลือก็ชวดเหมือนเดิม

**การแก้ปัญหาในรอบที่ 2:** ตอนนี้คงต้องงัดเทคนิคระดับ Senior ออกมาใช้: การสร้าง **"On-the-fly Variable"** เราจึงย้ายฟังก์ชันสุ่มมาไว้ในจังหวะของ request "Create Pet (03_POST)" เพื่อให้เกิดการสุ่มเลขชุดใหม่ ณ ตอนนั้นทันทีและเก็บลงตัวแปร `${CURRENT_PET_ID}` สิ่งนี้จะบังคับให้ user 100 คน มีการแรนดอม ID ใหม่ทุกครั้งเมื่อมีการสร้างข้อมูล

### บทที่ 3: ชัยชนะที่ไร้รอยตำหนิ (Run #3)

เราล้างไฟล์ผลลัพธ์เก่าทิ้งแล้วรันเทสต์ผ่าน Terminal เป็นครั้งที่ 3...

**ผลที่ได้:** Error Rate เป็น 0.00%! เทสต์นี้ผ่านหินด้วยจำนวนถึง 25,000 request โดยไม่มี `404 Not Found` โผล่มาแม้แต่ตัวเดียว หน้า Dashboard สว่างวาบไปด้วยตัวเลข PASS 100% รายงานผลดูสวยงามมาก 🎉.

![Test Results Summary](petstore-api-performance/test-results.gif)

**บทสรุปของระบบ:** นอกจากเราจะขจัด error ไปได้แล้ว สคริปต์นี้ยังปราศจากการชนของข้อมูลโดยสิ้นเชิง ทำให้เราสามารถเค้นศักยภาพและขีดจำกัดที่แท้จริงของระบบออกมาดูได้เต็มที่
