## 🌟 ทำไมผมถึงหยิบ Maestro เข้ามาอยู่ในกล่องเครื่องมือคู่ใจ

จริงๆ แล้วผมก็มีพื้นฐานใน **Appium** สำหรับฝั่งมือถืออยู่พอตัวแล้วครับ ซึ่งยอมรับว่า Appium นั้นทรงพลังมากๆ แต่มันก็แลกมาด้วยความยากในการเรียนรู้ที่ค่อนข้างสูง: ทั้งการพยายามเซ็ตอัปสภาพแวดล้อมที่แสนวุ่นวาย, ความเร็วในการรันเทสต์ที่ค่อนข้างอืดมืด, แถมบางทีพวกซอฟต์แวร์เบื้องหลังก็รวนแบบไร้สาเหตุอีก 

ผมก็เลยอยากลองหาทางเลือกใหม่ๆ ที่มันดูโมเดิร์น ยืดหยุ่น และเบาสบายกว่าเดิมครับ ซึ่งสิ่งที่ **Maestro** ให้มาก็คือ:

- **เชื่อมต่อปุ๊บ ติดปั๊บ**: ไม่ต้องมานั่งคอนฟิกตั้งค่า Driver ให้วุ่นวาย แค่เปิดอีมูเลเตอร์ขึ้นมา Maestro ก็กระโดดเข้าไปเชื่อมต่อได้ทันที!
- **UI ที่เป็นมิตรสุดๆ**: หน้าตาตอนเขียนเทสต์ไม่จำเป็นต้องดูน่าเบื่อขรึมๆ เสมอไป และ Maestro ก็พิสูจน์ข้อนี้ได้ดี!
- **ฟีลแบบ Low-Code**: มันใช้วิธีการเขียนโค้ดง่ายๆ แบบโครงสร้าง YAML แต่ลึกๆ แล้วเปิดให้เรายัดสคริปต์ JavaScript โหดๆ เข้าไปได้ด้วยนะ 

**ลองทอดสายตาดูพริ้วๆ ว่ามันเชื่อมต่อเข้ากับตัว Emulator ได้รวดเร็วขนาดไหน:**

### 📌 1. ระบบ Inspector ที่ติดมากับตัว ถือว่าเป็นเพชรเม็ดงามเลยล่ะ

จบปัญหาหาที่อยู่พิกัดปุ่ม (Locator) ไม่เจอเสียที! Maestro มาพร้อมระบบ Inspector ที่บิวต์อินอยู่ในตัวแบบดีงามสุดๆ เพียงแค่เราเอาเมาส์คลิกไปตามจุดต่างๆ บนหน้าแอปของคุณ มันก็สามารถเล็งประมวลผลให้ว่า Locator พิกัดแบบไหนที่มันเสถียรที่สุดในการหยิบเอาไปใช้ในโค้ดได้แบบอัตโนมัติเลยครับ

![Maestro Built-in Inspector](./assets/maestro-builtin-inspector.png)
*(ตัว UI Inspector ทำให้ประสบการณ์การนั่งงมหาเลนส์ปุ่ม กลายเป็นเหมือนเปิดระบบคลาสสิกลากและวาง Visual drag-and-drop ไปเลย)*

### 📌 2. การจัดการกับพวกหน้าต่างป๊อปอัพที่ไม่ได้รับเชิญ

พวกหน้าจอแอปมือถือมักจะชอบสุ่มเด้งเรทขอคะแนน, แวะให้พิมพ์อ่านจดหมายข่าว, ไม่ก็พวกข้อความแจ้งเตือนอัปเดตระบบเด้งขึ้นมาชนหน้าเราแบบแรนด้อมดื้อๆ 为了แก้เผ็ดปัญหานี้ ผมก็เลยงัดวิชาใช้ **เงื่อนไขจังหวะการไหล (Conditional Logic หรือ `runFlow`)** มาสู้ครับ
พูดง่ายๆ ก็คือ ผมสั่งสคริปต์ไปดื้อๆ เลยว่า: *"เห้ย ถ้าเอ็งเห็นป๊อปอัพนี่เด้งขึ้นมานะ แกก็แค่กดยกเลิกปัดทิ้งซะ ถ้าไม่เห็นก็รันงานที่เหลือต่อไปตามสบาย!"* จบครับ! ปัญหาที่ทำให้สคริปต์บั๊กพัง สลายหายไปในพริบตา

![Maestro RunFlow Conditional Logic](./assets/maestro-runflow-conditional-logic.png)
*(การหยิบโค้ด YAML มาใส่ลอจิกช่วยกรองพวกแบนเนอร์หรือความป่วนต่างๆ ให้ร่วงหายไปแบบสวยๆ)*

### 📌 3. ความแม่นยำระดับล็อกเป้าด้วย Relative Locators

สมมติว่าถ้าหน้าจอดันมีหน้าปุ่มข้อความซ้ำกันเป๊ะๆ โผล่มาสองปุ่มล่ะ เราจะทำไง? ผมลองใช้มุข **หาพิกัดแบบระบุพื้นที่อ้างอิงเทียบเคียง (Relative Locators)** เข้ามาสั่งการครับ โดยบอกเจาะจง Maestro ไปเลยว่า: *"เอ็งช่วยหาข้อความอันนี้นะ แต่ย้ำเลยว่าต้องเป็นส่วนที่วางเรียงกันอยู่ 'ด้านล่าง' ของอันนี้ และต้องอยู่ 'ด้านบน' ของอันนั้น"* งานนี้เราจะไม่มีปัญหากดปุ่มหลงทิศอีกต่อไปให้เวียนหัว!

![Maestro Relative Locators](./assets/maestro-relative-locators.png)
*(สามารถก้าวผ่านปัญหาชื่อปุ่มซ้ำซ้อนอย่างแยบยล ด้วยกลวิธีเทียบหาตัวชี้พิกัด)*

### 📌 4. ทรงพลังขึ้นอีกขั้นด้วย JavaScript ผ่านกระบวนการ `evalScript`

ในบางสถานการณ์ คุณก็มีความจำเป็นจะต้องกดบันทึกค่าเก็บลงตัวแปร, ก๊อปปี้เท็กซ์ข้อความในกล่อง, และเก็บไปเทียบผลลัพธ์ภายหลัง และข่าวดีก็คือ! Maestro เปิดโอกาสให้เราแทรกเส้นทางของโค้ด JavaScript สายเพียวอัดลงกลางวงโฟลว์งานได้ทันที!
ผมเลยเล่นใหญ่ใช้งานตัว `evalScript` ไปก๊อปข้อความจากอีกหน้าจอหนึ่ง ยัดฝากจำเข้าสมองของระบบไว้ แล้วเอาไปเทสเทียบเช็คผลตรงข้ามกับอีกหน้าจอนึงแบบไดนามิกซะเลย! ทั้งทรงพลังและยืดหยุ่นมากๆ ⚡

![Maestro EvalScript Copy & Compare](./assets/maestro-evalscript-copy-compare.png)
*(เล่นของหนัก โดยจับการเขียน YAML เข้าคูงัดเช็ค Validate logic เชิง JS ข้ามผ่านสลับหน้ากระดานจอ)*

## 🍏 ความท้าทายจากฝั่งระบบ iOS

การจะย้ายสคริปต์ทดสอบเทสไปสวมให้ฝั่งระบบ iOS ไม่ใช่งานง่ายๆ แค่ก็อปปี้วางแล้วเดินผ่านครับ พวกชื่อไอดี App IDs มันเปลี่ยนไปหมด แต่มันก็แลกมาด้วยข้อดีมหาศาล แล้วก็พ่วงแถมด้วยหลุมพรางสุดแสบนิดๆ ด้วยเหมือนกัน

**ข้อได้เปรียบมหึมา: โคตรลึกของ Locator IDs**
ระบบ iOS นิติตามธรรมชาติของมันคอยส่งชุดพิกัด IDs ชั้นยอดมาให้แปะควบคู่ไปกับวัตถุต่างๆ ให้เน้นๆ เลยครับ แทนที่เราจะพึ่งการยึดตัวพิมพ์ข้อความเป็นหลัก ผมสามารถเล็งยิงเจาะเข้าระบบไอดีแม่ได้ทันที ซึ่งในแง่มุมความเสถียรเนี่ย ถือว่าเป็นอันตายยากสุดๆ!

![Maestro iOS Locator ID](./assets/maestro-ios-locator-id.png)
*(เสพติดความแม่นยำของ iOS Locator IDs แบบไม่ต้องแคร์การเล็งจับข้อความแทนอีกเลย)*

**จุดหงุดหงิดพาปวดขมับ: ลูกกลิ้ง Native สไลด์หมุนของฝั่ง iOS**
ในขณะที่ดรอปดาวน์ธรรมดาคนอื่นใช้คลิกหา แต่พอเป็นมือถือฝั่ง iOS มันดันเปลี่ยนวงล้อสไลด์ทั้งหมดให้รวมเป็นกล่องอันใหญ่บึ้มเพียงก้อนเดียว เพื่อจะสั่งให้มันใช้งาน ผมสับสั่งคลิกหาข้อความไม่ได้ฮะ — เลยต้องจำยอมตั้งทิศคุม **พิกัดกว้างคูณสูง (X/Y Screen Coordinates)** เพื่อเลียนแบบวิถีจิ้มจอ รูดและสู้กลับด้วยมือคนเข้าจนได้! 👈👆

![Maestro iOS Scrollwheel Issue](./assets/maestro-ios-scrollwheel-issue.png)
*(หน้าตาลูกกลิ้ง Native iOS สุดดื้อด้าน ที่เหมารวมพฤติกรรมเป็นก้อนหินก้อนเดียว)*

![Maestro iOS Swipe Coordinates](./assets/maestro-ios-swipe-coordinate.png)
*(การล็อคพิกัดสั่งจำกัดทิศให้ระบบวิ่งรูดตีแตกเอาชนะเจ้ากงล้อดื้อตัวนั้นเสีย!)*

## 💡 ก่อนจะจากกัน

โปรเจกต์งานนี้ช่วยพิสูจน์ให้ผมเห็นกับตาเลยว่า เจ้า **Maestro** ถือเป็นการทุบกรงแหกกฏมิติใหม่ๆ ให้กับวงการ Automated สำหรับมือถืออย่างแท้จริง ด้วยการนำพวกเงื่อนไขโฟลว์ลอจิกเจ๋งๆ หรือลูกเล่นสอดสคริปต์ Javascript การวางตัวช่วยจับพิกัด Relative แบบเซ้นส์เหนือๆ รวมถึงการดัดเล็งพิกัดแบบเจาะจง ผมก็เลยคลอดระบบจำลองลื่นไหลเจ๋งๆ ที่วิ่งพล่านสลับพรมแดนค่ายผ่าน แอนดรอยด์ และ iOS หน้าตาเฉยออกมาได้ครับ 

บอกเลยว่าความรู้สึกที่ได้มา มันถือเป็นการก้าวกระโดดชิ้นมหึมา ให้กับพวกเฟรมเวิร์กโลกยุค Low-code แต่ในตินั้นมันก็ยังมีความยืดหยุ่นในการงัดกับ UI ป่วยซับซ้อนได้อย่างน่าทึ่งครับ

และสุดท้าย ขอบคุณสำหรับการติดสอยห้อยตามมาชม Maestro journey ห่ามๆ ของผมวันนี้นะครับ! 🚀
